## Executive Summary

P2300R10 (std::execution) and P2900R14 (Contracts for C++) are independently well-designed proposals that both serve demonstrated production needs. Ten of fifteen initial cross-coherence concerns were eliminated by the Challenge phase - three because they applied to any library (not P2300 specifically), three because they restate accepted 30-year C++ patterns (terminate bypasses everything; global replaceable functions are global), and three because the original analysis contained factual errors about P2900's mechanics. This pattern is itself significant: it suggests the two papers' interaction surface is narrower than initial inspection implies. Most apparent tensions dissolve on closer examination because the papers occupy genuinely different domains (protocol-level async composition vs. assertion-level correctness checking).

The five surviving findings cluster around a single structural theme: P2300's protocol-level requirements (noexcept receivers, move-through-channel efficiency, lazy composition) create contexts where P2900's contract features interact with measurable friction. Two findings reach high severity - enforcement overhead on hot paths cannot be locally opted out of (T-1), and adding preconditions to noexcept receiver functions inverts the safety calculus by introducing terminate paths that didn't exist before (T-2). Both are confirmed by independent committee awareness (P3166R0). These are not design defects in either paper individually; they are emergent properties of their composition that the committee should address with guidance or future mechanism.

## Compound Dynamics

Surviving compound dynamics from the stress test, ordered by severity.

### Contract Adoption Penalty (T-1 + T-2 + T-3)

Three independently-surviving findings that interact when a developer attempts to annotate P2300 sender/receiver code with contracts:

- T-1 imposes evaluation overhead on hot-path completion signals
- T-3 prevents postconditions from observing values without forcing copies
- T-2 makes preconditions on noexcept receivers potentially lethal

Interaction mechanisms:

- T-1 -> T-2: No per-site semantic disable means terminate risk cannot be mitigated locally
- T-3 -> T-1: Copy overhead from const params stacks on evaluation overhead in same code path
- T-2 -> T-1: Natural response ("accept overhead for safety") backfires - safety introduces danger

Navigated Precedent: Exceptions + noexcept + move semantics (C++11/17). Committee provided per-function opt-out (noexcept) and conditional move (move_if_noexcept). Key difference: contracts MVP has NO per-function semantic opt-out.

### Type-Erasure Gap (T-2 + T-4)

Virtual prohibition pushes contracts to non-virtual wrappers or body assertions. But receiver protocols are noexcept, so:

- T-4 forces contract checking behind the virtual interface
- T-2 means the concrete implementations (called through noexcept virtual dispatch) face terminate risk from contract_assert if predicate throws

Interaction mechanisms:

- T-4 -> T-2: Virtual prohibition forces contracts to noexcept NVI wrapper, directly triggering T-2's terminate risk
- T-2 -> T-4: NVI workaround (the "escape" from prohibition) enters T-2's danger zone

Navigated Precedent: Virtual destructors + noexcept (C++11). Committee accepted "don't throw in destructors" principle. Difference: destructor bodies are programmer-controlled; contract predicates can accidentally throw.

## Findings

Cross-coherence findings that survived the challenge phase, ordered by severity.

### [T-1] Contract enforcement overhead on P2300 hot paths
Severity: high | Confidence: high

A-9 defines zero-overhead composition under P2300's lazy sender model. A-13 defines the interaction between P2300's inherently-racy cancellation protocol and P2900's observable checkpoints (weakened: checkpoints only fire when evaluation actually occurs).

- Artifacts: A-9 (Zero-overhead under composition), A-13 (Cancellation races vs observable checkpoints, weakened)
- Papers: P2300R10, P2900R14
- Principles: Rule 1 (Zero Overhead), Rule 3 (Leave No Room Below), Rule 18 (Manual Control)
- Evidence: P3166R0 independently identifies this concern. P2900 design choice #3 (implementation-defined semantics). P2300's lazy composition model.

Enforce/observe semantics on P2300 hot-path customization points (set_value, start, connect, scheduler operations) reintroduce runtime checks at precisely the points P2300 eliminated overhead through lazy composition and intrusive storage. Observable checkpoints from contract evaluation act as optimization barriers, constraining reordering of cancellation-related code. The overhead is not opt-in at the composition boundary; it is inherited transitively through the sender algebra. P3166R0 independently identifies this concern: "One place where having accurate exception specifications is when using std::execution... An unnecessarily conservative noexcept(false) can result in additional overhead that the compiler cannot inline away."

Resolution probes:

1. Does P2300's bulk call set_value per-item or only on final completion? - Per-item calls go to the user function; set_value fires once per stage. But in deep pipelines (then|then|then|...), set_value fires per-stage-per-operation.
2. Can compilers optimize away trivially-true contracts under enforce? - Yes per P2900 design choice #11, but only when statically provable. Runtime-dependent predicates must be evaluated.
3. Does P2900 allow per-function semantic selection? - No. Semantic selection is implementation-defined and potentially global. No per-site opt-out.
4. Could a quality implementation avoid checkpoint overhead for proven predicates? - Yes. If evaluation is skipped entirely (proven result), there is no "beginning of evaluation" and no checkpoint. This weakens A-13's contribution but not A-9's direct overhead.
5. What is the expected deployment model for contracts on P2300 code? - P2900 Principle 5 says semantic should not be detectable at compile time. Libraries cannot defensively assume "ignore."

### [T-2] Noexcept receiver contracts create latent terminate (safety inversion)
Severity: high | Confidence: high

A-3 defines the interaction between P2300's protocol-required noexcept receiver completion functions and P2900's exception-from-predicate machinery.

- Artifacts: A-3 (Receiver noexcept vs exception-from-predicate)
- Papers: P2300R10, P2900R14
- Principles: Rule 5 (Visible Unsafety), Rule 8 (No Single Style), Rule 10 (Hybrid Styles), Rule 14 (Independent Composition), Rule 21 (Safe Easy, Unsafe Possible), Rule 23 (Local Verification), Rule 24 (Integration)
- Evidence: P3166R0 independently identifies this category. P2300 protocol requires noexcept on receiver completion functions. P2900 evaluation_exception mechanism.

P2300's receiver completion functions (set_value, set_error, set_stopped) are noexcept as a protocol requirement - not a user choice. If a contract precondition on a receiver's set_value has a predicate that throws during evaluation (e.g., calls a function that throws, dereferences something that triggers an exception), P2900 treats this as a contract violation with detection_mode::evaluation_exception. Under enforce/quick-enforce: program terminates. Under observe: handler called, and if handler returns normally, execution continues - but the original exception from the predicate is consumed by the contract machinery, not propagated. This creates a safety inversion: adding a contract (the "safe" thing) introduces a terminate path that did not exist before (making the code MORE dangerous). The programmer's intent was to add safety checking; the effect is latent process death from any predicate that can throw.

Resolution probes:

1. Can P2300 implementations make receivers non-noexcept? - No. Many sender adaptors (when_all, let_value internals) require noexcept completion functions for correctness. It is a protocol requirement.
2. Can predicates be constrained to noexcept expressions? - Not by P2900's design. Predicates are arbitrary boolean expressions. The evaluation_exception mechanism exists precisely because predicates CAN throw.
3. Does the observe semantic mitigate this? - Partially. Under observe, if handler returns, execution continues. The predicate's exception is consumed (not propagated out of the noexcept function). But the program state may be inconsistent.
4. Can static analysis catch potentially-throwing predicates? - Possible but not required by the standard. Quality implementations might warn.
5. Is there a pattern to write provably-noexcept predicates? - Yes: use only noexcept operations in predicates on noexcept functions. But this is undocumented guidance and easy to violate accidentally. No enforcement mechanism exists.

### [T-3] Postcondition const blocks value-channel moves
Severity: medium | Confidence: high

P2300's completion functions move values through channels; P2900 requires non-reference parameters odr-used in postconditions be declared const. If a sender adaptor's completion function has a postcondition referencing delivered values, those parameters must be const - preventing moves. Either the postcondition cannot reference the delivered values, or the values cannot be moved. The tension scales with type cost: int is unaffected, vector<T> pays a full copy. Workaround exists (contract_assert in body with subsequent move) but loses declarative postcondition semantics.

- Artifacts: A-14 (Parameter move semantics vs postcondition const)
- Papers: P2300R10, P2900R14
- Principles: Rule 1 (Zero Overhead), Rule 3 (Leave No Room Below), Rule 6 (Type Equality), Rule 9 (Features Over Prevention), Rule 14 (Independent Composition), Rule 21 (Safe Easy, Unsafe Possible), Rule 24 (Integration)

### [T-4] Virtual prohibition at type-erasure boundary
Severity: medium | Confidence: medium

P2300 acknowledges type erasure facilities will be built on top (e.g., any_sender<Sigs...>). These use virtual dispatch internally. P2900 makes it ill-formed to place pre/post on virtual functions. The NVI pattern and contract_assert in function bodies provide mitigations - severity is medium rather than high. Residual tension: existing non-NVI virtual interfaces require restructuring to gain contract coverage. P2900 Principle 14 explicitly leaves this "ill-formed to enable flexible evolution" - the limitation is temporal.

- Artifacts: A-5 (Virtual function prohibition vs type-erased senders)
- Papers: P2300R10, P2900R14
- Principles: Rule 4 (Static Type Safety), Rule 5 (Visible Unsafety), Rule 8 (No Single Style), Rule 10 (Hybrid Styles), Rule 12 (Transition Path), Rule 14 (Independent Composition), Rule 21 (Safe Easy), Rule 24 (Integration)

### Minor Findings

[T-5] Freestanding handler mechanism on GPU/accelerator targets (low severity, low confidence) - Quick-enforce resolves the GPU case; residual issue is governance (a domain cannot mandate its own semantic in source). Narrow practical concern.

## Positive Coherence

- Both papers target freestanding environments (A-12 alignment)
- Contracts invisible to concepts preserves P2300's type system (A-4)
- "Ignore" semantic provides genuine zero-overhead coexistence
- P2900's "no ABI break" and "no client-side break" protect P2300 code from involuntary contract effects
- The tombstone pattern (3 by Precedent) confirms that many apparent tensions are simply restatements of accepted C++ properties - the papers do not create novel hazards beyond those already inherent in the language
