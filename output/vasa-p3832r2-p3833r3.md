## Executive Summary

These two proposals exhibit strong structural coherence with the existing C++ standard library locking facilities. They follow established conventions (the `int`/`-1` return pattern from `std::try_lock`, RAII ownership from `unique_lock`, deadlock-avoidance delegation from `std::lock`) with disciplined extension into the timed domain. Six of eight initial findings were tombstoned by the Precedent test — the same structural tensions already exist in the C++11/17 locking facilities these proposals extend, and the committee accepted them there.

Two findings survive at medium and low severity. The primary concern is that `multi_lock`'s single `bool owns` member cannot represent partial lock states — a deliberate design choice that trades conceptual precision for API simplicity. The secondary concern is that multi-mutex preconditions scale linearly but remain equally invisible at the call site, an incremental extension of existing `unique_lock` risk rather than a qualitatively new hazard.

## Compound Dynamics

No compound dynamics survived the stress test. The two surviving findings (Rule 20 and Rule 23) are co-present in `multi_lock` but do not interact: the ownership representation (output) and the precondition visibility (input) operate on different sides of the API boundary.

## Findings

### [T-1] All-or-nothing ownership obscures partial-failure semantics
Severity: medium | Confidence: medium

`std::multi_lock<Ms...>` uses a single `bool owns` to represent ownership state across all held mutexes. After a `try_lock_for` returns a non-(-1) value indicating timeout, the programmer knows *which* mutex failed (from the return value) but the object itself remembers nothing — `owns_lock()` simply returns `false`.

- Artifacts: A-2 (multi_lock), A-3 (interaction)
- Papers: P3833R3
- Principles: Rule 20 (Direct Concept Mapping)
- Evidence: P3833R3 wording §32.6.X.3 ¶16: "owns is true if the return value equals -1, otherwise false"

The programmer's mental model of "I tried to lock three mutexes, two succeeded, one timed out" maps to a single `false` in `owns_lock()`. The return value carries the failure index, but the RAII object discards granularity. This is a deliberate simplification — per-mutex ownership tracking would complicate the class significantly — but it means `multi_lock` cannot represent the concept of "partial progress" that a programmer holding multiple `unique_lock` objects naturally has.

**Resolution Probe:** Does `multi_lock` support re-locking after a failed try? Yes — `defer_lock` construction + repeated `try_lock_for` is the intended pattern (Example 4 in P3833R3). This partially mitigates the tension: the single-bool design enables a clean retry loop. The tension is real but the design is self-consistent.

### Minor Findings

- **[T-2] Multi-mutex preconditions invisible at scale** (low severity, Rule 23): The precondition "no timed lock call while holding a lock on any argument" in P3832R2 scales with N mutexes while remaining equally invisible at the call site. This is incremental over existing `unique_lock(adopt_lock)` risk. Low confidence — the deadlock-avoidance algorithm is *designed* to manage this internally; the precondition primarily constrains implementers, not users.

## Positive Coherence

Strong positive signals across multiple principles:

- **Rule 17 (Deterministic Resources):** Textbook RAII with clear ownership semantics and exception safety at both layers.
- **Rule 24 (Integration, Not Isolation):** Both proposals satisfy existing named requirements (Cpp17BasicLockable, Cpp17Lockable, Cpp17TimedLockable) and compose with existing infrastructure (`scoped_lock(adopt_lock, ...)`, condition variables).
- **Rule 1 (Zero Overhead):** Pure template instantiation; no cost if unused.
- **Rule 11 (Real Problems):** Fills a concrete production gap — timed multi-lock has no current standard facility.
- **Rule 6 (Type Equality):** User-defined lockable types treated identically to standard mutex types.
- **Rule 18 (Manual Control):** Multiple escape hatches: `defer_lock`, `release()`, direct use of free functions without the RAII wrapper.

The 6 tombstoned findings (all killed by Precedent) confirm that these proposals faithfully extend existing C++ patterns rather than introducing novel structural risks.
